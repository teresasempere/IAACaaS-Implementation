IAACaaS

IoT Application-Scoped Access Control as a Service

Introduction

In the new Internet paradigm, where data economy is becoming a key element for companies success, along with the increasing concern of performing a sustainable development of their activities, the integration IoT devices in their ecosystems take special relevance. By 2020 the number of connected devices generating tons of sensitive data (sensors) or awaiting a request to trigger some reaction (actuators) will reach 26 billion. These devices usually have low computing capabilities and work over networks with limited connectivity. In this scenario, ensuring privacy, access control and identity management without implying a considerable decrement in performance, scalability or interoperability is really tricky.

Currently, there are several approaches to handle identity management and access control only for end users but not so many which also include IoT Systems management. Moreover, they do not cover all the requirements in terms of security or efficiency.

The aim of this project is to conduct an investigation on how can be implemented an IoT model that satisfies these requirements. This model, which is called IAACaaS (IoT Application-Scoped Access Control as a Service), is an access control approach for IoT scenarios based on OAuth2.0 authorization framework together with RBAC (Role Based Access Control) and ABAC (Attribute Based Access Control) authorization policies definition. This solution would allow not only the management of all IoT devices deployed in an ecosystem but also a flexible way of defining authorization policies to control requests from them as well as requests sent to them depends on who access to the IoT device and behalf of which service. It will be also studied how the inclusion of lightweight authentication and authorization mechanisms affect in terms of computational load, performance or even energy efficiency of devices and compare the results with other IoT security approaches.

Background

The GING (Next Generation Internet Group) of UPM (Polytechnic University of Madrid), in which this project will be carried out, has been directly involved with European projects related to identity management and access control for final users and IoT devices.

FIWARE is an open source platform that provides a series of components, known as Generic Enablers (GE), which ease the development of smart applications focused on the compilation and use of context information which is usually generated by IoT networks. Specifically, GING is responsible for several GEs:

‣ Identity Manager Keyrock → It is an OAuth2-based to manage access control and authentication solution for users and devices.
‣ PEP Proxy Wilma → Combined with Keyrock is used to control access to backend applications by checking defined authorization policies.
‣ Draco and Cygnus → These GEs are responsible for giving persistence to the history of context information received from the FIWARE Orion Context Broker.
‣ Cosmos → It is the GE for BigData analysis that can be integrated with other BigData platforms.

The projects in which GING has participated in the scope of FIWARE are: FI-WARE, FI-CORE and FI-NEXT.

The European regulation eIDAS (electronic IDentification, Authentication and trust Services) has as ultimate goal to offer the possibility to EU citizens to use their national eID in other EU countries when accessing public and private services online. GING is participating in several projects: CEF eID4U and CEF eID-FIWARE.

One of the main objectives of the projects is the integration of the FIWARE Identity Manager Keyrock with the CEF eID building block in order to allow CEF eID transnational authentication of EU citizens by means of their national eID in FIWARE based OAuth2 authentication domains. Specifically, in the scope of the CEF eID4U project, a number of educational services have been deployed, such as Moodle. Users profiles created with personal attributes retrieved from a eIDAS Node is enriched with academic attributes from universities, so that educational services can use all these data to particularize the educational offer for each student.

The integration of eID authentication in FIWARE components has opened new doors of research. Currently, GING is analysing how a user profile can be enriched with data regarding functional diversity. This way, applications and services could be adapted depending on the type of disability. This could mean, for instance, showing an ad-hoc web interface in an application or granting access to IoT networks in order to ease mobility in a physical space.

Description

IAACaaS, as it has been mentioned before, is a flexible and versatile access control model based on the OAuth2.0 protocol that eases the configuration of authorization policies for users and IoT devices. The OAuth2.0 protocol provides HTTP services with a delegated way of securing access to their resources for all the entities involved. In this case, these entities will be both users and IoT devices.

The main actors in the IAACaaS model are:

‣ Identity Provider. Manage users, services and devices and act as PAP (Policy Administration Point) and PDP (Policy Decision Point)
‣ Proxy. Act as PEP (Policy Enforcement Point) by intercepting requests from users and IoT devices
‣ Service Admin. Manage policies in the Identity provider.
‣ Services.
‣ IoT devices. Sensors and actuators)
‣ Users.

Due to the battery limitations of IoT devices, the main challenge is to transfer most of the logic needed to perform the authentication and authorization process out of them , so that a great part of the computation load is delegated to the PEP Proxy and the Identity Provider which are directly connected to the electrical grid.

In order to implement the proposed architecture, components from the FIWARE catalogue will be used. The three main components involved will be: Orion Context Broker , Identity Manager Keyrock and PEP Proxy Wilma. Orion will act as the backend of services, to which sensors will upload their measurements. These data will be available for users who fulfil the authorization policies of each service. Keyrock and Wilma will respectively play the roles of Identity Provider and PEP Proxy in the IAACaaS architecture. Keyrock already supports OAuth2.0 over HTTP and the definition of authorization policies for users. It will be extended to support the definition of policies for IoT devices, and the integration of the lightweight protocol COAP and its possibility to be used with OAuth2.0 over it will be studied.

On the other hand, in order to test the architecture, it is necessary to deploy different types of sensors that upload data or perform some kind of physical activity. To prototype these sensors and the communication between them and the FIWARE components, Arduino or Raspberries will be used.

Budget

This project has not been funded in the past. The main costs of this project are divided into personnel costs, dissemination, conferences and traveling and hardware equipment. As to personnel costs, it will be used to cover a researcher salary for one year and to offer a scholarship to involve a university student. Dissemination is the main purpose of this project, so publishing a research paper in an open access journal (two options are MDPI Sensors and IEEE Access) will help to transmit the results to the scientific community. Part of the funding will be used to pay for publishing in one of the journals mentioned. Besides, the funding will be used to cover travel costs in order to assist to meetings or conferences related to security in IoT. Finally, hardware equipment such as sensors or microcontrollers are needed to test the proposed architecture, so part of the funding will be used to buy these equipments.
